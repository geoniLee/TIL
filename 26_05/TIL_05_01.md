<h3>알고리즘 문제 풀이</h3>
<h4>4단 고음</h4>

* 알고리즘
    * 재귀 dfs / 백트래킹
    * 역방향 탐색

* 아이디어
    1. 1부터 시작해서 3씩 곱하여 최대 사용 가능한 mul의 수를 계산, 이를 바탕으로 사용 가능한 add의 개수를 산출
    2. 주어진 n 값을 기준으로 -1, /3을 해가며 남은 *, + 횟수가 0이 되고 n이 1이 되는 경우의 수의 개수를 반환
        1. +의 사용 가능 횟수가 *사용 가능 횟수 -1의 2배 이하면서 3의 배수 인 경우만 * 연산을 지울 수 있음

* 회고
    1. 문자열 생성
        1. 문자열 계산
            1. 최솟값 *++에서 시작해서 각 위치에 *++을 삽입하여 그 값에 해당하는 dp의 값을 1 올림
            2. 재귀로 생성된 문자로 들어갔을 때 map에서 해당 문자열을 확인하여 이미 확인한 값인 경우 바로 반환
            3. 각 문자열에 대해 매번 계산하여 값 산출
            4. dp[n]을 반환
            > 시간 초과 (연산과전 캐시를 해결 시도)
        2. 문자열 캐시
            1. 문자열을 front, *++, back으로 나누어 전달
            2. front+"*++"+back 이 처리되었는지 map으로 확인
            3. pair\<mul, add>로 저장된 map을 받아와 사용
                1. mul 이 0인 경우 문자열을 계산하여 map 값 갱신
                2. 0이 아닌 경우 * mul + add 연산
                3. 1 * front(2번 연산) * &'*++'(2번 연산) * back(2번 연산)을 값으로 구함
                4. 3번의 값을 사용해서 dp값에 +1
            4. dp[n] 반환
            > 시간 초과 (생성되는 문자열 수가 너무 많음)

<h3>언리얼 심화 라이브 세션</h3>
<h4>Templete Pattern & Sandbox Pattern</h4>

* Templete Pattern
    
    전체 실행 흐름은 부모쪽에서 고정하고, 세부 동작 일부만 자식 클래스 또는 Blueprint에서 변경할 수 있도록 하는 패턴

    * UFUNCTION(**, BlueprintNativeEvent)
        * BlueprintNativeEvent
            
            Cpp에서 기본 동작을 구현하고, 이후 Blueprint에서 필요하면 override 할 수 있는 함수

        * 실행 규칙
            * Blueprint 오버라이드가 없는 경우 -> Cpp의 함수이름_Implementation() 실행
            * Blueprint 오버라이드가 있는 경우 -> Blueprint 구현 실행
            * Blueprint에서 Cpp 기본 동작도 같이 실행 -> Parent 함수 호출

        * Cpp 구현 시 '함수 이름()'이 아닌 '함수 이름_Implementation()'을 구현

        * 호출할 때에는 '함수 이름_Implementation()'이 아닌 '함수 이름()'을 호출한다
            * '함수 이름()' 호출 -> Unreal이 Blueprint 오버라이드 여부를 확인 
            * 필요에 따라 Blueprint 구현 또는 Cpp _Implementation() 실행 

        * bp에서 사용 시
            * 우클릭 -> 부모 함수에 대한 호출 추가(Add Call to Parent Function)
            * parent 함수는 현재 blueprint의 부모 클래스에 있는 함수 구현을 호출함
                * 부모가 cpp 클래스 -> Cpp의 _Impletation() 실행
                * 부모가 blueprint 클래스 -> 부모 blueprint의 함수 구현 실행    
                    * 이 경우 Cpp까지 실행되려면 부모 blueprint 안에서도 다시 Parent 함수를 호출 해야함

        * 주의점
            * Cpp 함수 안에서 'Super::함수'를 먼저 호출하면, 부모 함수 내부에서 Blueprint Event를 호출하는 구조인 경우 Blueprint Event가 실행 될 수 있음
                * 이 경우 'Super::함수()' 아래에 작성한 Cpp 코드는 Blueprint Event보다 늦게 실행 될 수 있음
                * 따라서 Blueprint Event에서 바로 어떤 함수를 호출할 때 그 함수가 의존하는 Cpp 코드는 아직 실행되지 않은 상태일 수 있음

* Sandbox Pattern
    
    Cpp에서 기능 단위의 함수들을 제공하고, Blueprint에서 그 함수들을 조합해서 원하는 동작을 만드는 방식
    * UFUNCTION(BlueprintImplementableEvent)
        * BlueprintImplementableEvent

            Cpp에서 함수의 선언만 만들고 실제 구현은 Blueprint에서 작성하는 함수
        
        * 특징
            * C++ 기본 구현이 없음
            * Blueprint에서 구현하는 것을 전제로 함
            * BlueprintNativeEvent처럼 `함수이름_Implementation()`을 구현하지 않음
            * C++에서는 해당 함수를 호출할 수 있고, 실제 동작은 Blueprint 구현에 의해 결정됨

        * 실행 흐름

            ```text
            C++에서 함수 호출
            -> Blueprint에 구현된 이벤트 실행
            ```

        * 역할
            * Blueprint가 구현할 동작의 진입점


        * Blueprint에서 구현하지 않은 경우
            * 호출 자체는 가능하지만 실행될 Blueprint 내용이 없으므로 실질적인 동작은 없음
    
    * 장점

        * Blueprint에서 동작 흐름을 자유롭게 구성할 수 있음
        * C++ 코드를 수정하지 않고도 다양한 변형을 만들 수 있음
        * C++는 필요한 기능만 Blueprint에 노출할 수 있음
        * 프로그래머는 기본 기능을 제공하고, 디자이너는 Blueprint에서 조합하는 식으로 역할을 나누기 좋음

    * 주의점

        * Blueprint가 흐름을 직접 구성하므로 호출 순서에 대한 책임이 Blueprint 쪽에 생김
        * 필요한 단계를 누락하거나 순서를 잘못 배치하면 의도하지 않은 동작이 발생할 수 있음
        * 반드시 지켜야 하는 핵심 규칙은 C++ 쪽에서 방어적으로 처리하는 것이 안전함
        * 너무 많은 기능을 Blueprint에 노출하면 흐름이 복잡해지고 관리가 어려워질 수 있음
