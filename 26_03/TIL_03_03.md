<h3>C/C++</h3>
<h4>동적 할당</h4>

* 메모리 레이아웃 (스택 메모리, 힙 메모리, 코드 섹션, 데이터 섹션)
    - 코드 섹션: 작성한 소스코드들이 빌드된 결과물들이 저장되는 곳
    - 데이터 섹션: 전역 변수와 정적 변수 등이 저장되는 곳

    스택 메모리의 단점
    1. 런타임에 크기 조정 불가능
    2. 함수가 종료되면 해당 스택 프레임에 접근 불가능

    - 힙 메모리: 메모리 할당과 해제를 통해 관리되는 동적 할당 영역<br>
    **힙 메모리의 단점**
        1. 스택 메모리에 비해 할당/해제 속도가 느림
        2. 직접 메모리 할당 및 해제해야 합니다
    
    **동적 할당** <br>
    1. 메모리 할당(대여) - 반환된 값은 시작 메모리 주소
    2. 메모리 사용
    3. 메모리 해제(반납) - 메모리 주소를 돌려주지 않으면 메모리 누수가 발생함

    **동적 메모리 관련 함수**<br>
    
        할당:malloc()/calloc()
        재할당: realloc()
        해제:free()
        기타:memset()/memcpy()/memcmp()/...

    void* molloc(size_t size): size 바이트 만큼의 메모리를 반환/실패시 NULL

    void free(void* ptr): 동적 할당 받은 메모리를 해제하는 함수

    void* real1oc(void* ptr, size_t new_size): 메모리 시작 주소 ptr을 new_size 바이트로 재할당 해주는 함수

<h3>Cpp 코딩 테스트 완전 정복</h3>

<h4>vector vs deque</h4>

* vector: 선형 컨테이너<br>
특정 데이터에 접근하기 위해서는 해당 데이터의 인덱스를 통해 접근<br>
    
    * 데이터가 N개 일 때, 특정 원소를 찾는 연산 (O(N))

* deque: 맨 앞의 원소와 맨 뒤의 원소를 가리키는 포인터가 있고, 내부적으로 데이터는 여러개의 청크(chuck)라고 불리는 영역으로 쪼개져 있음

* 맨 앞에 원소를 추가하는 경우
    * Vector => 뒤에 있는 원소들이 전부 이동(O(N))
    * deque => 청크를 나눠서 관리하므로 다른 원소들이 이동할 필요가 없음O(1)

* 임의 접근을 하는 경우
    * vector => O(N)
    * deque => map을 내부 적으로 관리함 (O(N)) / 데이터가 커질 경우 map을 통해 위치를 찾고, 접근하는데 연산횟수가 증가함

<h4>특정 원소를 찾는 경우</h4>

* 컨테이너의 내부 정렬
    - 레드-블랙 트리로 관리
    - 원소 삽입/ 삭제 / 탐색 모두 O(logN)

* set & map : 데이터를 레드-블랙 트리로 관리 / 특정 원소를 찾는 연산은 O(logN)

* unordered_set & unordered_map : 해시 기반으로 데이터를 관리 / 특정 원소를 찾는 연산은 O(1)

<h4>문자열을 결합하는 경우</h4>

* '+'연산자 -> 새로운 문자열을 다시 만듦(기존 문자열 길이가 N이라면 O(N))
* '+=' 연산자와 append() 메서드 -> 새로운 문자열을 만들지 않고, 기존 문자열에 덧붙임 (O(N))

<h4>auto& vs auto</h4>

* auto -> 복사가 발생
* auto& -> 복사가 발생하지 않음
