* 알고리즘
    * BFS
    * 그래프 탐색

* 아이디어
    1. 일반적인 BFS에 빙판 처리 추가
    2. while문으로 빙판은 산이나 끝에 닿을때까지 반복해서 이동
    3. 방문 정보 처리할 때 vector<bool>은 특수화 되어 있어, vector<int>을 사용하므로 이를 활용해서 방문정보로 0~3으로 관리

* 회고
    1. 빙판의 경우 가로로 들어가는 경우와 세로로 들어가는 경우가 다르기 때문에 여러번 들어갈 수 있도록, 방문 처리를 하지 않음
        1. 특정 상황에서 빙판이 반복적으로 queue에 들어가며 메모리 초과 발생 
        2. 미방문 0, 가로로 진입 1, 세로로 진입 2, 가로/세로 모두 진입 완료 && 초원의 방문 으로 변경

    2. 빙판의 방문 처리 미흡으로 인한 queue.push 누락
        1. 빙판으로 인해 방문하는 모든 지역에 방문 정보 증가
        2. 방문 정보가 정확한 값을 가지지 못해 방문해야 했으에도 방문하지 못하는 지역 발생
        3. 빙판의 시작 지점과 끝 지점만 방문 정보 값 갱신

<h3>게임 개발자를 위한 Cpp 문법</h3>
<h4>디자인 패턴</h4>

* 디자인 패턴<br>
개발 시 반복적으로 등장하는 문제를 해결하기 위한 일반화된 솔루션
    1. 생성 패턴: 새로운 것을 만들어내는 방법과 관련된 패턴
    2. 구조 패턴: 여러 부품을 어떻게 조립하고, 연결하는 방법에 대한 패턴
    3. 행동 패턴: 부품들이 서로 어떻게 상호작용할지에 대한 패턴

* 싱글톤 패턴: 특정 클래스의 인스턴스를 오직 하나만 생성하여 재사용하는 생성 디자인 패턴
    * 생성 및 대입을 private로 구현
    * getInstance()를 통해서만 인스턴스를 받음

* 데코레이터 패턴: 기존 객체를 수정하지 않고도 새로운 기능을 추가하거나 변경할 수 있도록 하는 패턴이다.
    * 데코레이터 클래스: 피자 객체를 감싸며, 동적으로 새로운 기능을 제공합니다

    **예시**

    ```
    Pizza* p = new BasicPizza();
    p = new CheeseDecorator();
    p = new PepperoniDecorator();
    ```
    **결과**
    > 1. BasicPiazza 반환 => "Basic Pizza"
    > 2. CheeseDecorator 반환 => "Basic Pizza + Cheese"
    > 3. PepperoniDecorator 반환 => "Basic Pizza + Cheese + Pepperoni"

* 옵저버 패턴: 객체의 상태 변화를 관찰하는 관찰자들을 주체에 등록하여, 상태 변화 시마다 메서드를 통해 직접 통지하는 패턴
    * Subject: 상태를 관리하고, 변경되었음을 옵저버에게 알립니다
    * Observer: Subject을 관찰하며, 상태 변경 시 반응합니다.
    * 연결구조 -> Observer들은 Subject에 등록하고 변경이 발생하며 알람을 받습니다.
