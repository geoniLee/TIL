<h3>제어 흐름 구조 선택 기준</h3>

제공받은 강의 코드 리뷰 중 Arrow Anti-Pattern으로 보일 수 있는 코드가 발견되었고, 이를 Early Return으로 변환하여 구현하던 중 제한 사항을 발견하여 관련 내용 조사하기 시작함

* Arrow Anti-Pattern

    중첩 조건문이 깊어져 코드 모양이 화살표처럼 꺾이는 구조

    * 단점
        * 조건이 깊어질 수록 가독성이 떨어지고, 핵심 처리가 어디있는지 찾기 어려워짐
        * 중간 조건을 추가하거나 예외처리를 넣을 때 유지보수가 어려워짐
        * 조건 분기가 많아져 코드 복잡도가 증가한다
        * 에러처리와 실패 흐름이 명확하게 드러나지 않음
        ---
        * 특히 Unreal Engine에서는 자체 API 등을 호출 시 기본적으로 길이가 긴 경우가 있어 depth이 적어도 크게 체감될 위험이 있음

* Early Return

    함수 초반에 실패 조건이나 예외 조건을 먼저 검사하고, 조건을 만족하지 못하면 즉시 Return하여 함수를 종료하는 방식

    * 장점
        * 들여쓰기 깊이를 낮출 수 있음
        * 실패 조건이 함수 초반에 드러남
        * 가독성이 좋아짐
        * 조건 추가나 수정 시 중첩위치를 찾는 부담이 줄어듦
    
    * 선택하지 않은 이유
        * 하나의 함수 안에서 여러 작업이 독립적으로 수행되는 경우, 앞쪽 작업의 실패를 이유로 함수 전체를 종료하면 뒤쪽의 작업까지 실행되지 않을 수 있음

* 새로운 함수 or 람다 함수

    특정 작업 단위를 별도의 함수나 람다 함수로 분리하는 방식

    * 선택하지 않은 이유
        * 쟁점이 되는 코드는 초기화 시점 1회만 사용되는 코드이므로, 재사용 가능성이 낮은 코드를 위해 함수를 만드는 것은 과하다고 판단함

* Do-While
    Do- While 형태로 코드를 감싸서, 실패 조건에서 break를 사용해 빠져나오는 방식

    * 선택하지 않은 이유
        * 실제로 반복이 필요한 코드가 아님에도 반복문 형태를 사용해야함

* goto

    특정 label의 위치로 제어 흐름을 직접 이동하는 방식

    * 사용을 결정한 이유
        * Early Return처럼 실패 조건에서 빠르게 작업을 벗어날 수 있음
        * 이동 범위가 짧고 이동 목적이 명확함
        * 현재 상황에서는 goto의 일반적인 단점보다 제어 흐름 보존의 장점이 더 크다고 판단함
