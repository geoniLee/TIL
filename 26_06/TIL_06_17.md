<h3>알고리즘 문제 풀이</h3>
<h4>디스크 컨트롤러</h4>

* 알고리즘
    * 정렬
    * 우선순위 큐
    * 그리디

* 아이디어
    1. 요청 시간을 기준으로 보기 위해 정렬
    2. 첫번째 인덱스의 요청 및 같은 시점에 요청한 작업까지 pq에 push
        * 작업 시간, 요청 시점, 인덱스의 tuple로 넣어 차례로 우선순위를 갖도록 함
    3. 작업 완료 시점을 현재 시간으로 갱신하고, 현재 시간 - 작업 요청 시간을 answer에 합산
    4. 현재 시간보다 빠른 시점의 모든 작업을 pq에 push
        * 현재 idx의 작업이 현재 시간보다 늦을 경우
            1. pq가 비어 있음-> 현재 시간을 해당 idx의 시간으로 초기화 하고 push
            2. pq가 비어있지 않은 경우 -> push하지 않고 넘어감
    5. answer을 jobs의 크기로 나눈 값을 반환

<h4>애니메이션 동기화</h4>

* 애니메이션 동기화
    * CharacterMovementComponent가 관리하는 이동 정보는 캐릭터 이동 복제를 통해 서버와 클라이언트 사이에 동기화됨
    * CharacterMovementComponent가 관리하지 않는 별도 변수 Server RPC로 서버에 전달하고, Property Replication을 통해 다른 클라이언트에 동기화 해야함

<h4>공격 동기화</h4>

* 공격 동기화
    * 공격 애니메이션은 자동으로 복제되지 않으므로 각 클라이언트에서 재생되도록 별도로 동기화 해야함
    * 모든 처리를 서버 응답 이후 실행하면, 네트워크 지연만큼 입력 반응이 늦어짐
    * 로컬 클라이언트는 공격 애니메이션을 먼저 재생하고, 서버는 요청을 검증한 뒤 다른 클라이언트에 공격 표현을 전달하는 방식으로 지연을 줄일 수 있음

* 공격한 클라이언트
    > 로컬에서 공격을 즉시 실행<br>
    > Server RPC로 공격 요청 전달

* 서버
    > 공격 가능 여부와 게임 결과 검증<br>
    > 다른 클라이언트에 공격 전달<br>
    > 피격 및 데미지 최종 처리

* 주의 사항
    * 서버의 확인이 필요한 기능은 Server RPC 사용(ex. 공격 요청)
    * 여러 클라이언트에 공격 표현을 전달할 때는 NetMulticast RPC or 대상별 Client RPC 사용
    * 애니메이션처럼 다음 상태로 보정할 수 있는 시각적 표현은 Unreliable로 처리할 수 있음
    * 데미지처럼 중요한 게임 상태는 서버에서 처리하고 검증해야함
